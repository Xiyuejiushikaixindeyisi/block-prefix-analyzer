# Prefix Cache 自动分析与可视化模块 — 阶段一实施计划

> 本文件由设计阶段产出，用于阶段一开发期间持续参考。所有设计决策已与用户确认。
> 阶段一只实现"输入 model_name → 生成模型级报告"。阶段二（部门维度）仅做 schema 预留。

---

## 0. 项目背景

`block-prefix-analyzer` 当前是一个离线分析库，每个分析模块（F4 / F9 / F10 / F13 / F14 / e1 / e1b / reuse_rank / reuse_distance / common_prefix）独立产出 CSV+JSON+PNG。

**数据来源**：MaaS 平台采集，单次窗口 = 2 小时（2026-04-24 9:00-11:00 为基准批次）；私有数据，CSV 形式存放于另一台机器。CSV 列：`{chat_id, user_id, raw_prompt, timestamp(相对采样起点偏移)}`。

**最终形态（路线图）**：分析器自动采样模型 30 min 数据 → 离线分析 → 卡片式展示给用户「该部门 / 该用户使用了哪些模型，每个模型的理想命中率 / 流量业务模式 / KV cache 时间局部性」，让客户直观感受我们提供的优化与定制服务。

**阶段一目标**：在已有分析能力之上，新增一个**自动聚合 + Streamlit 可视化**模块，让非命令行用户输入模型名即可看到完整的 Prefix Cache 分析报告。

**阶段二（不在本计划内）**：输入 `department_name`，输出该部门下 `user_id × model_id` 的报告。

## 1. 已确认的设计决策（12 条）

| # | 决策 | 关键含义 |
|---|---|---|
| 1 | **Streamlit 部署形态** | 阶段一仅供开发者本地 `streamlit run` 自用；不做刷新按钮、不做权限。Dashboard 只读 `outputs/maas/<model>/report.json`，显示 `generated_at` + `data_version`。 |
| 2 | **report.json 生成时机** | 阶段一手动运行 `build_model_report.py`。等流程稳定后再在 `run_maas_analysis.sh` 加可选参数 `--with-report`，**不**默认强制。`build_model_report.py` 也**不**自动调用 converter / 生成 single_turn 子集；这些是上游手动步骤。 |
| 3 | **content_type_guess 类别** | 仅做 KV cache 优化视角的粗分类，类别固定为：`system_prompt` / `rag_template` / `agent_tool_prompt` / `json_schema` / `code` / `qa_template` / `long_document` / `other`。Regex + 启发式即可，不引入 transformers / sklearn / langdetect。 |
| 4 | **建议条目语气** | `priority: P0/P1/P2` + `confidence: high/medium/low`。每条必须含 `conclusion / evidence / action`。**口径风险（如 reuse_time_p50=0、request_interval_p50=0）作为 Warning 类型输出，不包装成强优化建议。** |
| 5 | **block_size sweep 口径** | §1 的 "block_size sensitivity" 用 e1_user_hit_rate 四档 sweep `[16, 32, 64, 128]`。**短中长上下文（8k–32k）必须 sweep**；64k+ 长上下文模型若 sweep 跑不动，YAML 可降级为单档 `128`，dashboard 优雅降级显示"单档结果"。F4 / F13 / F14 / reuse_distance 沿用各自配置的主 block_size（128）。页面需明确说明两者口径不同。 |
| 6 | **依赖策略** | `streamlit` + `pandas` 放进 `[ui]` extras。核心 `src/analysis/` + `report_builder` 保持标准库或现有依赖。pandas 只在 dashboard 层用。 |
| 7 | **Phase 2 数据形态** | 业务侧将提供 CSV 映射，字段：`department_id, department_name, user_id, model_id`。**阶段一只预留 schema 字段**，不写 CSV loader。若日志本身含 department 字段则优先用日志；否则未来用 CSV join。 |
| 8 | **离线无 UI 模式** | `build_model_report.py` 必须可独立运行并生成 report.json。Streamlit 只是 view，不承担分析/聚合。 |
| 9 | **单轮 / 多轮口径** | **`turn_index == 0` 为单轮，`turn_index > 0` 为多轮**。预过滤为 `requests_single_turn.jsonl`，喂给 F13；F14 用全量 `requests.jsonl`（其内部仅处理多轮）。**不修改 F13 / F14 分析模块**——通过预过滤输入实现口径切换。 |
| 10 | **F9 / F10 归入 §2** | F9（会话轮数 CDF）、F10（用户级轮数均值/标准差 Lorenz）作为 §2"流量业务模式 → 会话结构"子节。 |
| 11 | **§4 内容来源** | top-N 共识块 + decoded_text 来自 **common_prefix** 输出（不跑 e5_block_text，因为 64k+ 长文本需 2h+，性能不可接受）。content_type_guess 应用于 common_prefix 的 decoded_text。 |
| 12 | **Reusable 变体不做** | F4 / F13 / F14 dashboard 只展示 `content_prefix_reuse` 变体。Reusable 变体（`content_block_reuse_anywhere`）是广义 reuse，**非 vLLM APC hit**；业界 cacheblend 等非前缀方案目前仍是实验阶段，无工程落地。reusable YAML 命令行可保留但 dashboard 不展示。 |

## 4. content_type_guess 规则（v1）

输入：common_prefix 的 `decoded_text`（按 ConsensusBlock 切片后的文本）；输出：8 类标签之一。

按以下顺序判定（首次命中即返回）：

| 优先级 | 类别 | 判据 |
|---|---|---|
| 1 | `json_schema` | 含 `"type": "object"` / `"properties"` / `"required":` JSON Schema 结构特征 |
| 2 | `agent_tool_prompt` | 含 `"type": "function"` 或 `"name":` + `"description":` + `"parameters":` JSON 结构 |
| 3 | `system_prompt` | "你是" / "You are" / "Your task" 开头，长度 > 200 字符 |
| 4 | `rag_template` | 含 "参考资料" / "上下文" / "Context:" / "Reference:" / "Documents:" 关键词 |
| 5 | `code` | 非空白字符中 `{};()[]=<>` 占比 > 8%，或匹配 `def \|class \|function \|import \|public \|private ` |
| 6 | `qa_template` | 同时含 ("问题" \| "Q:" \| "Question:") + ("答案" \| "A:" \| "Answer:") |
| 7 | `long_document` | 长度 > 1000 字符且未命中以上 |
| 8 | `other` | fallback |

实现位置：`src/block_prefix_analyzer/analysis/content_classifier.py`

## 5. recommendation 规则（v1）

每条规则一个 Python 函数，签名 `(report: dict) -> Optional[Recommendation]`。

| 规则 ID | 触发条件 | 输出 |
|---|---|---|
| `R-PIN-CHAIN` | common_prefix `prefix_length_blocks ≥ 32` 且首块 `coverage_pct ≥ 60%` 且 ideal_hit ∈ [0.4, 0.7] | P0 high recommendation：pin 核心 chain |
| `R-CAPACITY-FIRST` | ideal_hit ≥ 0.7 且 working_set_60min > 100K | P1 high：先做容量梯度实验 |
| `R-LOW-CEILING` | ideal_hit < 0.30 | P2 high：不建议优先做淘汰算法，转向租户隔离 / RAG cache |
| `R-BATCH-TTL` | reuse_time p50 ≤ 1s 且 p80 ≤ 5s 且 ideal_hit ≥ 0.4 | P1 medium：batch 级短 TTL 保护 |
| `R-LONG-TTL` | reuse_time p80 > 30s 且 ideal_hit > 0.5 | P1 medium：试 TTL-LRU，base_ttl = p80 |
| `R-MULTI-TENANT` | F10 lorenz top10% 用户贡献 turns 占比 > 60% 且 F9 mean_turns > 3 | P2 medium：多租户 chain 竞争，做租户分区 |
| `R-CACHE-PRESSURE` | reuse_distance p80 > available_cache_blocks（若已配置）；或 reuse_distance p80 > 50000 兜底 | P1 high：reuse 窗口长 + 容量小，淘汰快于命中，建议分级存储或更大容量 |
| `W-SAME-SECOND` | request_interval_p50 == 0 | **Warning**：理想命中率可能虚高 |
| `W-REUSE-ZERO` | reuse_time_p50 == 0 | **Warning**：reuse 多发生在同一秒，需 batch 验证 |

实现位置：
- `src/block_prefix_analyzer/recommendation/rules.py`（每条规则一个函数）
- `src/block_prefix_analyzer/recommendation/engine.py`（注册 + 跑全部规则 + 排序输出）

## 9. 阶段一明确不做的事

| 不做 | 原因 |
|---|---|
| Streamlit 内现算分析 | 性能 + 稳定性都不允许 |
| 多模型对比视图 | 阶段一单模型 |
| Plotly 交互图 | 先用 `st.line_chart` |
| 实时刷新 / 文件监听 | 手动重跑 build_model_report 即可 |
| 数据库 / 缓存层 | JSON 足够 |
| 用户认证 | 本地工具 |
| 自动重跑（数据变更检测） | data_version 显示出来即可，重跑由人决定 |
| pandas 进 src 核心 | 仅在 dashboard 层用 |
| 多语言 / PDF 导出 | 不在范围 |
| Phase 2 的 CSV 映射加载器 | 等映射文件到位 |
| 修改任何已有 analysis 模块 | 风险大于收益 |
| `run_maas_analysis.sh` 默认调用 build_model_report | 阶段一手动跑；流程稳定后再加可选参数 |
| F4 / F13 / F14 reusable 变体进 dashboard | 业界非 prefix cache 仅实验；cacheblend 等无工程落地 |
| e5_block_text 跑长上下文 | 64k+ 需 2h+，性能不可接受；用 common_prefix 替代 |
| 修改 F13 / F14 内部单/多轮判定 | 通过预过滤输入实现口径切换，不动模块代码 |
